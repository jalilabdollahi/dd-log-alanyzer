🧠 Detection & AI Analysis

Four detection algorithms + AI-powered anomaly interpretation via AWS Bedrock

📐

Z-Score Volume Anomaly

Logs are bucketed into 5-minute windows. The Z-score formula (x - μ) / σ flags any window exceeding 2.5 standard deviations from the mean — catching sudden spikes or unusual drops.

💥

Error Burst Detection

A sliding window of 60 seconds scans error logs. When 10+ errors concentrate within a single window, it triggers a burst alert — catching rapid failure cascades across services.

🔗

Error Correlation

Errors are fingerprinted by tokenizing messages (stripping IDs, IPs, timestamps). Similar errors cluster together, tracking affected services and identifying root-cause candidates by earliest occurrence.

📈

Trend Analysis

Linear regression tracks the direction of log volumes — increasing, decreasing, or stable. Compares the current window against the historical baseline to catch gradual degradation.

🤖

AI Anomaly Interpreter

Every detected anomaly is sent to AWS Bedrock (Claude 3 Haiku) for AI-powered interpretation. The model analyzes raw metrics, sample error messages, and service context to generate a root cause hypothesis, impact assessment, and recommended investigation steps — delivered directly in Slack and Jira.

🚀 Key Capabilities

Built for production environments processing millions of logs daily

⚡

High Volume Ready

Tier 1 aggregation covers 2M+ logs server-side. Only 5K are downloaded for deep analysis.

🔄

Dual Deployment

Run ad-hoc via CLI or deploy as AWS Lambda for 24/7 automated monitoring.

🎯

Preset Queries & Service Discovery

Use pre-configured presets (kong-errors, api-5xx) or use the --all-services flag to automatically discover and analyze every active service.

📊

Smart S3 Reports

Generates rich HTML & JSON reports, uploads to S3 with pre-signed URLs, and skips presets with zero findings to save storage.

🔕

Multi-Service Deduplication

Smart fingerprinting (SQLite locally, DynamoDB on Lambda) prevents alert fatigue while ensuring unique alerts fire for simultaneous cross-service anomalies.

🏥

Maintenance Health Check

Every watch cycle hits a configurable maintenance API. Unhealthy services trigger per-service Datadog drilldowns — with automatic Kong Gateway attribution when no errors are found in DD.

⌨️ CLI Commands

All commands share the same analysis engine — run ad-hoc or automate

Command Description Key Flags
dd-logs health Verify Datadog API + Slack + Jira connectivity and list active presets —
dd-logs query Search and display raw logs from Datadog --time --limit --preset
dd-logs analyze Two-tier full analysis: patterns, anomalies, errors, trends. Supports Slack & Jira notifications --notify-slack --create-jira --format html|json --output
dd-logs report Generate a standalone HTML or JSON report file without console output --format html|json --output <path> --preset
dd-logs watch Continuous monitoring loop: maintenance API check (Phase 1) + anomaly analysis (Phase 2) every interval --interval --all-services --maintenance-url --no-slack --no-jira
Example — Error analysis
❯ dd-logs analyze \
  "env:prod service:konggateway" \
  --time "last 1h" \
  --notify-slack \
  --create-jira
Example — HTML report
❯ dd-logs report \
  "env:prod" \
  --time "last 24h" \
  --format html \
  --output report.html
Example — 24/7 watch mode
❯ dd-logs watch \
  --all-services \
  --interval 60 \
  --maintenance-url \
  "https://api.prod.../maintenance"
