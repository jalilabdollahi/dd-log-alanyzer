Analysis Engine

Two-tier log analysis

Breadth + depth — aggregate coverage meets sampled deep-dive analysis.

Tier 1 · Server-side
Aggregation
  • Covers ALL logs — no sampling
  • Datadog metrics aggregation API
  • Volume, error rate, service breakdown
  • Fast — handles massive datasets
  • Produces trend & volume signals
⊕
merged
Tier 2 · Targeted
Deep Analysis
  • Sampled log content analysis
  • Fingerprint clustering & patterns
  • Anomaly Z-score + burst detection
  • Cross-service error correlation
  • Raw log context for AI interpreter
🎯
Combined result: Tier 1 gives population-level statistics across every log line. Tier 2 provides semantic understanding of what is failing and why. Together they produce accurate anomaly scores with actionable root-cause context — eliminating both false positives and blind spots.
AI-Powered

AI Anomaly Interpreter

AWS Bedrock · Claude 3 Haiku — turns raw error logs into DevOps incident briefs.

1
Anomaly detected
Analysis engine surfaces a Z-score spike or error-burst pattern across one or more services.
2
All error logs sent to Bedrock
The actual raw error-level log messages from the anomaly window are passed to Claude 3 Haiku.
3
DevOps-grade brief generated
AI produces a structured incident description covering what failed, root cause hypothesis, impact, and investigation steps.
4
Attached to Slack & Jira
AI description is embedded in Slack Block Kit messages and Jira ticket bodies. Graceful fallback if Bedrock unavailable.
💰 ~$0.0001 per anomaly · ~$0.25 / M input tokens
🤖 Sample AI Output · Claude 3 Haiku

What is failing: aggregatoradapter is returning connection timeout errors to downstream consumers at 3× baseline rate. Pattern: Failed to connect: read timeout after 30s

Root cause hypothesis: The upstream corecustomeragreement service appears degraded — 80% of errors originate within 200ms of its API calls, suggesting database or network saturation.

Impact: ~340 affected requests in the last 15 minutes. Customer agreement fetches failing silently in 4 downstream services.

Investigation: Check kubectl top pods -n prod for corecustomeragreement. Review Datadog APM traces for DB slow queries. Verify RDS connection pool limits.

Option A · CLI

Local CLI reference

Six commands covering every workflow — from quick queries to 24/7 watching.

🔌 Health check
dd-logs health
🔎 Query logs
dd-logs query "env:prod status:error" -t "last 30m" -l 200 dd-logs query --preset kong-errors -t "last 6h"
📊 Generate report
dd-logs report "env:prod" -t "last 24h" \ --output daily.html
🧪 Analyze (full pipeline)
dd-logs analyze "env:prod" -t "last 1h" \ --notify-slack --create-jira dd-logs analyze --preset api-5xx \ -f html -o report.html --notify-slack
🔭 Continuous watch
dd-logs watch --all-services \ --interval 300 dd-logs watch "service:konggateway" \ -i 60 --no-jira

Getting Started

Up and running in minutes

Three paths — pick your mode and go.

CLI (Local)
1
Install
pip install -e .
2
Configure
cp .env.example .env # Add DD_API_KEY, DD_APP_KEY, # SLACK_WEBHOOK_URL, JIRA creds
3
Run
dd-logs health dd-logs analyze "env:prod" -t "last 1h" dd-logs watch --all-services
Web Dashboard
1
Install + run
pip install -e ".[web]" cd webapp/frontend && npm install # Terminal 1: dd-logs-web # Terminal 2: npm run dev # → http://localhost:5173
AWS Lambda (24/7)
1
Store secrets
aws secretsmanager create-secret \ --name dd-log-analyzer/secrets \ --secret-string '{ "DD_API_KEY": "...", "DD_APP_KEY": "...", "SLACK_WEBHOOK_URL": "..." }'
2
Store YAML config
aws ssm put-parameter \ --name /dd-log-analyzer/config \ --value "$(cat config/default.yaml)"
3
Build & deploy
bash scripts/package.sh cd infra terraform init -backend-config="bucket=..." terraform apply
